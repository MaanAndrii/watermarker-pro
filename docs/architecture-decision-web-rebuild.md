# ADR: Чи варто робити переробку з нуля на Streamlit?

## Контекст

Поточний проєкт — інструмент пакетної обробки фото з вотермарками.
Ціль: стабільний вебдодаток для роботи в локальній мережі (та потенційно далі).

## Рішення

**Якщо переробляти “з нуля” під повноцінний вебдодаток, я б не обирав Streamlit як основну платформу.**

- Для **MVP/внутрішнього інструменту** (1-5 користувачів, швидкий запуск): Streamlit ок.
- Для **продуктового вебдодатка** (рольова модель, черги, масштабування, API, фонові задачі):
  краще **FastAPI + Celery/RQ + Redis + React/Next.js** (або SvelteKit/Vue).

## Чому не Streamlit як long-term база

1. **Архітектурні обмеження UI/runtime**
   - модель rerun і websocket-орієнтована взаємодія;
   - складніше контролювати великі uploads/довгі процеси, фонові задачі та retry-політики.

2. **Складніший production hardening**
   - auth/roles/audit/logical multi-tenant краще лягають на класичний backend API.

3. **Front-end UX гнучкість**
   - кастомні workflow (черги, історія задач, сторінки статусів, permissions) швидше ростуть на SPA/SSR стеку.

## Рекомендований стек для “з нуля”

- **Backend API**: FastAPI
- **Worker**: Celery або RQ
- **Broker/Cache**: Redis
- **Storage**: локальний диск/NAS + PostgreSQL (метадані)
- **Frontend**: Next.js (або React + Vite)
- **Reverse proxy**: Caddy/Nginx
- **Auth**: Keycloak/Auth0/Authelia (залежно від LAN/SSO вимог)

## Мінімальний roadmap міграції

1. Винести image-processing ядро у чистий сервісний шар (без Streamlit-залежностей).
2. Додати REST API для створення задачі обробки.
3. Додати фонового worker-а для batch-job execution.
4. Реалізувати web UI для черги/прогресу/завантаження результатів.
5. Додати auth, quotas, audit log, обмеження ресурсів на рівні worker.

## Коли Streamlit все ще виправданий

- Single-tenant LAN інструмент.
- Немає вимоги до складної ролі/аудиту/API.
- Потрібна максимальна швидкість розробки і підтримка невеликою командою.

## Підсумок

**Відповідь:** для повного “з нуля” вебпродукту — **ні, не Streamlit як основа**.
Streamlit лишив би тільки для прототипу/адмінської панелі або внутрішнього tooling.
